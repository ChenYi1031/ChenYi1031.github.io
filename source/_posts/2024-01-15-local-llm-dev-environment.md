# 本地大模型开发环境搭建：Ollama 与 LM Studio 的选择

## 为什么要做本地环境\n\n很多学习与验证场景并不需要把数据发到云端：例如在本地做 Prompt 迭代、写小工具、做 RAG 原型。一个稳定的本地环境，价值在于可复现、可离线、成本可控。\n\n## 两种路线：Ollama vs LM Studio\n\n- **Ollama** 更像是『本地推理服务』：提供命令行、拉取模型、启动 HTTP API，方便集成到脚本/后端。\n- **LM Studio** 更像是『桌面客户端』：模型下载管理、对话 UI 更友好，也能开本地服务，适合快速试用。\n\n## 我推荐的组合\n\n1. 用 **Ollama** 作为统一的推理入口（有 API，方便后续接入你的博客小工具或自动化脚本）。\n2. 用 **LM Studio** 做可视化对比与快速试错（看 token/s、显存占用、温度等参数差异）。\n\n## 常见踩坑与排查\n\n- 输出乱码：确保终端编码 UTF-8；Windows 下建议使用新版终端。\n- 性能不稳定：确认是否启用 GPU 加速，模型量化等级是否合适。\n- 上下文太短：优先看模型本身上下文限制，其次看推理端参数（例如 max_tokens）。\n\n## 最小可用的本地 API 调用示例\n\n`ash\n# 以本地服务为例，检查是否能连通\ncurl http://localhost:11434/api/tags\n`\n\n后续你做 RAG 或写测试脚本时，可以把这个 API 当作『可替换的模型后端』。
