# API 自动化测试架构：分层、数据驱动与可观测性

## 先解决三件事\n\n1. 用例是否稳定\n2. 失败是否好定位\n3. 维护成本是否可控\n\n## 分层建议\n\n- client：HTTP 请求封装（重试、超时、鉴权）\n- service：业务动作封装（登录/下单/查询）\n- 	est：断言与场景编排\n\n这样可以避免每个用例里到处复制粘贴请求细节。\n\n## 数据驱动\n\n不要把大量测试数据写死在代码里。你可以用 YAML/JSON 管理输入，再用 fixture 生成随机但可追踪的数据（例如带 trace_id）。\n\n## 可观测性\n\n把关键字段打印出来：equest_id、核心入参、响应码、耗时。\n失败时优先让人能一眼知道：是网络问题、鉴权问题、还是业务回归。
